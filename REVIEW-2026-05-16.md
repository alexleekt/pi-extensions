## 🔴 Critical Issues

### 1. `shared/` and `types/` missing from `package.json` `files` array
**File:** `package.json:44-53`

The `files` array controls what gets published to npm:
```json
"files": [
    "index.ts",
    "tool",
    "util",
    "fallback",
    "dist",
    "README.md",
    "LICENSE"
]
```

**Missing:** `shared/` and `types/`.

**Impact:** `tool/ask-user.ts` imports `../shared/ask-user.js`. `fallback/terminal-prompt.ts` also imports from `../shared/ask-user.js`. `types/glimpseui.d.ts` is in `tsconfig.json` `include`. If these directories aren't in the tarball, the extension will fail at runtime with module-not-found errors after `npm install`.

**Fix:**
```json
"files": [
    "index.ts",
    "tool",
    "util",
    "fallback",
    "shared",
    "types",
    "dist",
    "README.md",
    "LICENSE"
]
```

### 2. `flatOptionsFallback` treats single-select-without-freeform as freeform
**File:** `fallback/terminal-prompt.ts:91`

```ts
if (options.length === 0 || (!allowMultiple && !allowFreeform)) {
    // Single text input
```

If `options.length > 0`, `allowMultiple = false`, and `allowFreeform = false`, the condition `(!allowMultiple && !allowFreeform)` is `true`, so it shows a plain text input instead of the options list. The user would never see the choices they provided.

**Fix:** Remove the `(!allowMultiple && !allowFreeform)` condition. It only makes sense when `options.length === 0`:
```ts
if (options.length === 0) {
    const text = await ui.input(question);
    ...
}
```

Note: `allowFreeform` defaults to `true` in `index.ts`, so this bug may not manifest in typical usage. But if a caller explicitly sets `allowFreeform: false` with `allowMultiple: false` and provides options, the fallback is broken.

### 3. `smoke-test.ts` listens for non-existent `"ready"` event
**File:** `scripts/smoke-test.ts:33-35`

```ts
win.on("ready", (info) => {
    console.log("✅ WebView ready", ...);
});
```

The `glimpseui.d.ts` declares only `"message"`, `"closed"`, `"error"` events on `GlimpseWindow`. There is no `"ready"` event. This callback will never fire, and depending on the glimpseui implementation, it might throw or silently fail.

**Fix:** Remove the `"ready"` listener. The smoke test already auto-closes after 2 seconds and uses `"message"` and `"error"` for actual interaction.

### 4. `visual-qa.ts` "Questionnaire (freeform)" scenario uses wrong payload shape
**File:** `scripts/visual-qa.ts:94-115`

The scenario named "Questionnaire (freeform)" sets `type: "questionnaire"` but provides an `options` array instead of a `questions` array:
```ts
payload: {
    type: "questionnaire" as const,
    options: [
        { title: "Timeline", description: "..." },
        ...
    ],
}
```

The `questionnaire` type expects `questions: Question[]`. `getQuestions()` returns `payload.questions ?? []`, so this scenario renders a questionnaire dialog with **zero questions** — just an empty dialog with a title and context.

**Fix:** Change `options` to `questions` and structure each as a freeform `Question` (no `options` field, just `title` and `description`):
```ts
questions: [
    { title: "Timeline", description: "What's your target launch date?" },
    { title: "Budget", description: "Approximate budget range (USD)" },
    ...
]
```

### 5. `safe()` uses `any` types, defeating TypeScript
**File:** `util/safe-callback.ts:10-27`

```ts
export function safe<T extends (...args: any[]) => any>(
    label: string,
    fn: T,
): T {
    return ((...args: any[]) => {
        ...
    }) as T;
}
```

The `any[]` parameter types and return type erase all type information. Callers get no compile-time checking that the wrapped function is being called with the right arguments. The `as T` cast at the end is also unsafe — it claims the wrapper returns the same type as `fn`, but the wrapper actually returns `undefined` on sync errors.

**Fix:** Use proper generics with constrained parameter types:
```ts
export function safe<Args extends unknown[], R>(
    label: string,
    fn: (...args: Args) => R,
): (...args: Args) => R | undefined {
    return (...args) => {
        try {
            const result = fn(...args);
            if (result && typeof (result as Promise<unknown>).then === "function") {
                (result as Promise<unknown>).catch((err: unknown) => {
                    console.error(`[pi-ask-user-glimpse] ${label} async error:`, err);
                });
            }
            return result;
        } catch (err) {
            console.error(`[pi-ask-user-glimpse] ${label} error:`, err);
            return undefined;
        }
    };
}
```

### 6. `summarizeTitle` stopword list has stemmed forms that don't match after punctuation stripping
**File:** `tool/ask-user.ts:38-42`

The stopword list includes "don", "doesn", "didn", "won", "wouldn", etc. But the `summarizeTitle` function strips punctuation with `/[^\w\s]/g` before splitting. So "don't" becomes "dont", not "don". These stemmed stopwords will never match and just bloat the set.

**Fix:** Either remove these stemmed forms (they serve no purpose), or replace the punctuation stripping with smarter tokenization that handles apostrophes in contractions.

### 7. `optionRefs` arrays grow unbounded and aren't cleaned up
**Files:** `webview/src/components/SingleSelect.tsx:19`, `webview/src/components/MultiSelect.tsx:18`

```ts
const optionRefs = useRef<(HTMLButtonElement | null)[]>([]);
```

The ref callback `ref={(el) => { optionRefs.current[idx] = el; }}` sets entries but never removes them. When the search filter changes, old entries at indices that no longer exist remain in the array, pointing to detached DOM nodes. While this is mostly harmless (detached nodes won't throw on `focus()` or `scrollIntoView()`), it's a memory leak pattern and could cause confusion during debugging.

**Fix:** Use a `Map` keyed by option title (like Questionnaire does), or clean up the array when `filtered` changes:
```ts
useEffect(() => {
    optionRefs.current = [];
}, [filtered.length]);
```

### 8. Questionnaire keyboard navigation lacks Arrow key support
**File:** `webview/src/components/Questionnaire.tsx`

Unlike SingleSelect and MultiSelect which have full ArrowUp/ArrowDown navigation with roving tabindex, Questionnaire buttons have no Arrow key navigation. Users can Tab through them, but there's no way to navigate between options within a question using Arrow keys. This is an accessibility gap.

**Fix:** Add ArrowUp/ArrowDown handlers similar to SingleSelect/MultiSelect, scoped per-question.

### 9. `displayMode` warning fires for any truthy value, including valid enum string
**File:** `tool/ask-user.ts:119-122`

```ts
if (params.displayMode) {
    console.warn("[pi-ask-user-glimpse] displayMode parameter is ignored...");
}
```

The check is `if (params.displayMode)` which fires for `"overlay"`, `"inline"`, or any other string. But `displayMode` is typed as `StringEnum(["overlay", "inline"])`, so the user is providing a VALID value that happens to be ignored. A warning about an ignored valid parameter is confusing.

**Fix:** Only warn if `displayMode` is an unexpected value, or remove the warning entirely since the parameter is documented as legacy/ignored in the schema description.

### 10. `Enter` in MultiSelect submits instead of toggling focused option
**File:** `webview/src/components/MultiSelect.tsx:110-114`

```ts
} else if (e.key === "Enter") {
    e.preventDefault();
    if (selected.size > 0) {
        handleSubmit();
    }
}
```

When a user is navigating options with Arrow keys and presses Enter, it submits the form. In SingleSelect, Enter selects the focused option. This behavioral inconsistency is documented in keyboard shortcuts but could surprise users who expect Enter to toggle (like Space does).

**Recommendation:** Either make Enter toggle the focused option (like Space) and only submit when Ctrl+Enter is pressed, or document this behavior more prominently.

### 11. `response-formatter.ts` produces empty-string selection on malformed input
**File:** `tool/response-formatter.ts:55-58`

```ts
selections: Array.isArray(result.selections)
    ? result.selections.map(String)
    : [String(result.selection ?? "")],
```

If `result.selections` is not an array and `result.selection` is undefined, this produces `[""]`. The text output would be an empty string, which is confusing.

**Fix:** Add a guard:
```ts
const selection = result.selection ?? "";
if (!selection) {
    return { content: [{ type: "text", text: "No selection" }], ... };
}
return [String(selection)];
```

## 🟢 Low Issues / Suggestions

### 12. `freeform` button label inconsistency between SingleSelect and MultiSelect
**Files:** `webview/src/components/SingleSelect.tsx:144`, `webview/src/components/MultiSelect.tsx:99`

SingleSelect shows: `Other: "{query || "..."}"` (with quotes)
MultiSelect shows: `Other: {query || "..."}` (without quotes)

Minor UI inconsistency.

### 13. `Questionnaire` buttons lack `tabIndex` management
**File:** `webview/src/components/Questionnaire.tsx`

All buttons use the default `tabIndex=0`, meaning Tab navigates through EVERY button on the page. With many questions and options, this is tedious. Consider adding `tabIndex={-1}` to non-focused options and managing focus like SingleSelect/MultiSelect do.

### 14. `detect-conflict.ts` error message string matching is fragile
**File:** `util/detect-conflict.ts:8-10`

```ts
function isNotInitializedError(err: unknown): boolean {
    return err instanceof Error && err.message.includes("Extension runtime not initialized");
}
```

If Pi changes this error message, the conflict detection silently breaks. Consider matching on a custom error code or type instead.

### 15. `glimpseui.d.ts` `GlimpseWindow.on()` uses method overloads instead of discriminated union
**File:** `types/glimpseui.d.ts`

The current type:
```ts
on(event: "message", handler: (data: unknown) => void): void;
on(event: "closed", handler: () => void): void;
```

This is fine but could be more extensible with a discriminated union type for events.

### 16. `validate.ts` placeholder check is too broad
**File:** `scripts/validate.ts:28-33`

```ts
if (html.includes("ASK_USER_PAYLOAD")) {
    console.log("✅ HTML contains payload placeholder");
}
```

This matches any occurrence of the substring, including in comments or unrelated text. Should check for the exact placeholder `/*ASK_USER_PAYLOAD*/`.

### 17. `STOPWORDS` set is duplicated between `tool/ask-user.ts` and itself
The set includes both full forms and stemmed forms, plus words like "s", "t" (single letters) that are unlikely to appear as standalone words. Consider deduplication and removal of unnecessary entries.

### 18. No unit tests
The project has validation scripts (`validate.ts`, `smoke-test.ts`, `visual-qa.ts`) but no automated unit tests. Consider adding tests for:
- `summarizeTitle` edge cases
- `formatResponse` with all response kinds
- `allAnswered` logic in Questionnaire
- Payload injection escaping

### 19. `activeIndex` not reset when search filter changes but length stays same
**File:** `webview/src/components/SingleSelect.tsx:47-49`

```ts
useEffect(() => {
    setActiveIndex(-1);
}, [filtered.length]);
```

If a search replaces option A at index 0 with option B at index 0, the length stays the same and `activeIndex` isn't reset. The focus ring would appear on a different option than expected. Very minor UX issue.

### 20. `window.glimpse.send` type assertion repeated 12+ times
**Files:** All webview components

The pattern `(window as unknown as { glimpse: { send: (data: unknown) => void } }).glimpse.send(...)` is repeated everywhere. Extract to a helper:
```ts
function sendToGlimpse(data: unknown): void {
    (window as unknown as { glimpse: { send: (data: unknown) => void } }).glimpse.send(data);
}
```
